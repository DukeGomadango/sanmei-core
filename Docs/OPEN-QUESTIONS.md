# 確定待ち論点（OPEN QUESTIONS）

以下は主に**占い師／監修者と仕様書または真理値表で確定**すべき項目です。エンジンは `rulesetVersion` とセットで固定します。各節の **（採用: …）** はプロダクト／アーキテクチャ方針として先に固定したもので、番号末尾の**監修で詰めること**のサブ箇条書きが残タスクです。**暦・時刻の 1・4 は sanmei-core v1 として本文で固定済み**（以降の変更はマイナー版以上の見直しとする）。

## 暦・時刻

1. **節入り日・月柱境界**（**採用: sanmei-core v1**）  
   「節入りの瞬間」は天文学的な一瞬（マスタでは分〜秒単位）が存在する。算命学コアは**時柱を出さない**ため API は問診に依る「どちらの月生まれか」の推測はできず、**時刻比較で機械的に決める**か、**情報不足ならエラーにする**必要がある。  
   **数学的境界**（同一 `context.timeZone` の民用時刻系）: 出生のインスタントを \(t\)、当該「節」の入りインスタントを \(t_s\) とする。\(t < t_s\) なら**前月の月柱**、\(t \ge t_s\) なら**当月の月柱**（節入りの瞬時は当月側に含める）。サブ秒の扱いは実装で固定するが、**半開区間ではなく「以上で当月」**を正とする。  
   **時刻必須の条件（v1: ±0 日）**: `user.birthDate` が、マスタ上の当該節の入りインスタントを **`context.timeZone` に投影した暦日**（**節入り日**）と**一致する**場合に限り、`birthTime` を**必須**とする。判定の実装手順は **（1）節入りの絶対瞬間（例 UTC）を取得 →（2）`context.timeZone` でローカル化し暦日 `YYYY-MM-DD` を抽出 →（3）`birthDate` と文字列完全一致**に固定する（**同一 UTC でも TZ によりローカル日が前後する**ため、`birthDate` だけを別経路で組み立てて比較しない）。それ以外の暦日では `birthTime` 未指定でも**日単位**で月柱を確定してよい（[REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) §5 の日柱日界に従う）。  
   **エラー**: 上記に該当するのに `birthTime` が欠ける場合は計算を行わず **HTTP 422**、`code`: `TIME_REQUIRED_FOR_SOLAR_TERM`（[REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) §7）。  
   **監修・製品で詰め残すこと**: UX 上「節入り前後±1 日はウィザードで時刻入力を促す」などは**クライアント施策**としてよいが、**v1 API の 422 契約は「節入り日一致」時のみ**とする。文献・手計算とのゴールデンは `rulesetVersion` 単位で蓄積する。  
   **蔵干用「深さ」の暦定義**（ローカル暦日の JDN 差・月柱と同一基準節）については [IMPLEMENTATION.md](./IMPLEMENTATION.md) §5.0 を正とする。
2. **タイムゾーン**: `context.timeZone` に **IANA 名（例: `Asia/Tokyo`）を必須**とする方針を推奨（暦・時刻 **1・4** と整合）。第1フェーズを JST 固定 UI にしても、API 契約は TZ を明示する。
3. **夏時間（DST）**: 対象地域を限定するか、TzDB を用いたライブラリに委ねるか。`birthDate`／`birthTime` は**その TZ の壁時計時刻**として解釈される（暦・時刻 **4**）。
4. **民用標準時と真太陽時の責務**（**採用: sanmei-core v1**）  
   四柱推命では時柱のために出生地経度を用いた真太陽時（地方時）補正が中心になるが、算命学コアは**時柱を生成しない**ため、`birthTime` は**節入り境界**にしか使わない。実務・ソフトでは、厳密に真太陽時で切る派と、例として日本国内では **JST のまま**国立天文台等の節入り表の時刻と比較する簡易派が併存する。  
   **API v1 の固定**: `sanmei-core` は `user.birthDate` および `user.birthTime`（指定時）を **`context.timeZone` における民用標準時（壁時計時刻）**として解釈する。節入りマスタの各インスタントは**UTC 等の絶対瞬間**として保持し、比較時はリクエストのローカル日時と**同一タイムライン上**で対比する（サーバーは**均時差・経度による真太陽時変換を行わない**）。  
   **クライアント側の上乗せ**: 監修・学派が「真太陽時で境界を切る」を要請する場合は、**呼び出し前**に補正した時刻を `birthTime` として渡す（日付を跨ぐなら `birthDate` も合わせて調整する）。API は「渡された民用（またはクライアントが自称する補正後）時刻」とマスタ瞬間の比較機である。  
   **ゴールデン**: 再現性は **同一ペイロード＋`rulesetVersion`** で固定。経度補正をクライアントに寄せた分、**テストフィクスチャは「入力として渡す birthDate/birthTime の意味」をコメントまたはフィールドで明示**する。  
   **フィールド**: `birthLongitude` / `birthCityCode` は **v1 の節入り・蔵干計算では参照しない**（任意。将来拡張・表示・別機能用）。詳細は [REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) §5。

## 学派（`sect`）ごとの差分

5. **学派差の実装マトリクス**: 正統算命学では月柱はいずれも節入りで切る。人体図・大運の差は主に **(A) 蔵干切替の日数テーブル** と **(B) 立運の日数÷3 の端数・起算** による。各 `sect` の具体値を表に落とす（[DOMAIN-GLOSSARY.md](./DOMAIN-GLOSSARY.md)「学派差」）。
6. **蔵干（初元・中元・本元）**: 節入りからの経過日（深さ）と採用ルールを学派ごとに定義。
7. **大運**: 立運年齢の**端数処理**、順逆行、節入り日生まれの例外（日数の数え方）。
8. **天中殺の「スライド」と API の責務境界**（**採用方針の骨子**）  
   - **条件の外部化**: スライド成立条件はハードコードせず、**JSON またはドメイン向け DSL**で `ruleset` に版付きで保持する（無限 `if` 回避）。  
   - **機械部分は API が完結**: API コアは当該 `ruleset` を解釈し、**ゴールデンテスト可能な出力**（例: 機械的に確定する候補期間・上限ポテンシャル、`isShifted` 相当、スコア）を返す。**占断の数値・フラグに効く機械的条件の再適用をフロント専用ロジックに置かない**（実装二重化とテスト分裂を防ぐ）。  
   - **鑑定実務との差**: 問診・生活状況に依る**ナラティブな「実際に稼働しているか」の説明**は API が持たず、フロント／LLM に委ねてよい。ただし **API が返した機械的结果と矛盾する上書き**（別の `isShifted` をクライアントだけで決める等）は行わない。  
   - **監修で確定すること**: 上記 DSL／ルール表の真理値、レスポンスに載せるフィールドの最小セット。[REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) の `tenchuSatsuStatus` 説明および [SECT-RULESET-MATRIX.template.md](./SECT-RULESET-MATRIX.template.md) の `isShifted` 行と整合させる。
9. **虚気（`Kyoki`）**: 干合・月支条件・`shadowYousen` の生成条件を学派別に列挙。
10. **格法・位相法**: `allowGohouInKaku` が効く範囲（半会・支合・方三位等の一覧と優先度）。
11. **位相法の拡張集合**: `research` で扱う候補（干合・納音・律音・天剋地冲・大半会）を `kind` と `sourceLevel` 付きで分離管理するか。
12. **位相法の適用スコープ**: 命式内判定と、命式×大運/年運のクロス判定をどの層（Layer3a/3b）で処理するか。
13. **高尾系一次 locator の回収運用**: 拡張位相法ごとに `volume/chapter/page/quotation` を必須化し、`L0_PRIMARY_VERIFIED` 昇格条件にする。
14. **locator 進捗管理**: `PENDING-LOCATOR` のまま長期滞留しないよう、`owner/due/lastChecked` を最低メタとして持つ。
15. **priority 運用（P0/P1）**: `P0` は実装ブロッカー、`P1` は改善追跡。`P0` は `lastChecked` が14日以上空いたら再探索を起票する。  
    **拡張位相法の初期割当（Iteration 24）**: `P0` は `TENKOKUCHICHU` / `DAIHANKAI` のみ。`P1` は `KANGO` / `NACHION` / `RITSUON`。

## データ表現

11. **六親法と位置情報（スロットポインタ）**  
    干のみのマップでは、算命学の六親読みで必要な **「その干／星が命式のどこにあるか」**（天干か、どの柱の蔵干か、初元・中元・本元のどれか）が失われる。配偶者・母などが**日支蔵干にあるか年干にあるか**等で意味づけが変わる説明がある。  
    **（採用: スキーマ方針）** 六親ノードには **柱・蔵干スロット等の座標**を必ず含める。監修でフィールド名・列挙子の細部を調整する余地は残すが、**干だけ**の返却を Layer2 の基準形としない。フロント／人体図ハイライトとの整合のため。[REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) §6.2 `familyNodes` を正本とし、HTTP 例のフラットマップは廃止。  
    **確定すること（監修）**: レスポンス例は次のような**座標付き**を起点とする:  
    `[{ "role": "MOTHER", "stem": "癸", "location": { "pillar": "MONTH", "slot": "ZOUKAN_HONGEN" } }]`  
    監修者と **JSON スキーマ**で最終合意する。
12. **守護神・忌神**: 参照するマトリクスの版（文献／学派）の固定。

## L2c（`energyData`／`destinyBugs`）— 採用済み骨子と監修タスク

以下は**コード未実装でもドキュメント上固定した**方針である（詳細は [IMPLEMENTATION.md](./IMPLEMENTATION.md) §2「Phase L2c」、[REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) §6.2）。

| 論点 | 採用 |
|------|------|
| `energyData` の入力 | **位相法・虚気を含めない**素の三柱＋蔵干。位相後の数理は将来 Layer3（例: `shadowEnergyData`）。 |
| `energyData` の幾何・丸め | API は**極座標（角度）＋面積比**に正規化、**固定丸め**でゴールデン安定。 |
| `destinyBugs` | **出生で凍結・静的のみ**。年運／大運天中殺・スライドは `dynamicTimeline`。 |
| mock ruleset | **`mock-v1.json` を主系**とし必要に応じ拡張。検証用に `mock-internal-v2` 等を追加可。`meta.schemaRevision` で世代追跡。 |

**監修または実装直前に詰めるとよいこと**

- `destinyBugs` の **`code` 最終名**と教材用語の対応（プレースホルダ: `SHUKUMEI_TENCHUSATSU_YEAR`、`SHUKUMEI_TENCHUSATSU_MONTH`、`IJOU_KANSHI_NORMAL`、`IJOU_KANSHI_DARK`＝**暗干支**）。
- **`SHUKUMEI_TENCHUSATSU_MONTH`** を製品に載せるか（学説・教材により要否が分かれうる）。
- **`energyData` の丸め**を常にエンジン固定にするか、`ruleset` に逃がすか（学派差の有無）。
- **五行スケールの高次補正**（例: 気図法の 178 スケール、季節補正・従星点の反映）を `takao-v1` 等でどの ruleset 契約に切り出すか。現行は `energyByElement`（mock 重み集計）までを第一段階として固定。

## 検証データ

13. **ゴールデンマスター**（**採用: 早期 CI**）  
    開発初期から、監修者と合意した**学派別の正解ペア**（入力 JSON／期待出力 JSON）を **最低でも数十件規模**（高尾・朱学院など `sect` ごと）リポジトリ管理し、**CI で回すテストハーネス**を用意する。Rust 移行は**同一フィクスチャへの一致**を完了条件とする（[ARCHITECTURE-AND-CONTRACTS.md](./ARCHITECTURE-AND-CONTRACTS.md) §6）。  
    **（併行: モック ruleset）** **内部整合ゴールデン**（`mock-v1` 等の**非監修** `rulesetVersion` に対し、「当該 JSON のルールを追うとこの出力になる」期待値）を別系統で CI に載せてよい。外部サイトとの流派差でテストが不安定になる問題を避ける。監修ゴールデンが揃い次第、同じハーネスでディレクトリまたは `rulesetVersion` だけ分岐する。  
    **監修で詰めること**: 最低件数の最終値、カテゴリ別の厚み（節入り境界・スライド疑義・閏月等）、合意済みサンプルの増やし方。
