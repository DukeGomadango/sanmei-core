# ドメイン用語・技法辞書（DOMAIN GLOSSARY）

本書は算命学コア API を設計するための**概念境界・語彙・データの型**を整理したドメイン辞書である。Web 上の公開情報・一般命理に基づく**共通枠**と、製品が採用する**可変ルール**を区別する。**四柱推命の用語・四柱（時柱を含む）のモデルをそのまま持ち込まない**こと（ルーツは共通でも用語体系と命式の次元が異なる）。混同しやすい点は各節で明示する。

| 正本の所在 | 内容 |
|------------|------|
| 本書（GLOSSARY） | 層構造、用語、普遍に近い定数、算出**パターン**、型の形 |
| [REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) | HTTP 契約、`asOf`、エラー、Priority Resolver、レスポンス塊 |
| [IMPLEMENTATION.md](./IMPLEMENTATION.md) | リポジトリ内 **実装の現状**（ファイルマップ・節入りパイプライン・**経過日数 §5.0**・Orchestrator 方針）。GLOSSARY の「型」に対応するコード位置の索引 |
| `ruleset` / 監修文書 | 学派別の**真理値表・行列・IF 文・数式**（蔵干採用、星マトリクス、数理式、格法条件など） |
| [OPEN-QUESTIONS.md](./OPEN-QUESTIONS.md) | 監修で確める細部（スライド真理値表等）。**採用方針**（例: sanmei-core v1 の民用時／節入り境界、スライド条件の DSL 化）は同文書の各節に記載 |
| [SECT-RULESET-MATRIX.template.md](./SECT-RULESET-MATRIX.template.md) | `sect` × ルール項目の表 |

## ドキュメント構造：3 層アーキテクチャ

実装の依存の向きは原則 **`Layer 1 → Layer 2 → Layer 3`**（上位層は下位を参照する）。

| 層 | 英名 | 役割 |
|----|------|------|
| **Layer 1** | **Primitives** | 陰陽・五行・干支・干合ペアなど、API 上の**列挙子と関係テーブル**（命理で広く共有される基盤）。変更はバージョン付きで慎重に。 |
| **Layer 2** | **Core Models** | **生まれた瞬間（＋採用ルール）で凍る**陰占・陽占・数理・六親・格法フラグなど。入力は主に `user`＋`sect`＋`rulesetVersion`。 |
| **Layer 3** | **Dynamic Interactions** | **`asOf`・大運・流年流月**、位相法、虚気、天中殺稼働、Resolver 結果など時間軸・結合で変わる量。 |

**注意**: Layer 1 も「表示ラベル」や一部命理流派の別表は揺れうる。算命学固有の**蔵干採用ルール・蔵干日数テーブル・大運立運の端数処理・星表**は **Layer 2 以降の `ruleset` データ**としてバンドルし、本書には**スロット構造と参照先**だけを書く。**月柱の節入り境界そのもの**は正統な算命学各派で共通（§「学派差（算命学における本質的差分）」）。

## Layer 0：算命学の位置づけ（文脈）

- 日本で体系化された占術の一つ。**陰占**（十干十二支の器）と**陽占**（人体図上の星）を組み合わせる説明が一般的。
- **朱学院**・**高尾学館**等、学び舎・流派が複数存在する。人体図・大運などの差異の本質は §「学派差（算命学における本質的差分）」に集約する（**「節入りを使わない算命学」という分岐は正統系には存在しない**という前提で設計する）。

## Layer 1：Primitives（基礎定数と型）

### 1.1 陰陽（YinYang）

API では列挙子（例: `YIN` / `YANG`）で表す。表示名は i18n。

### 1.2 五行（Gogyo）

API では列挙子（例: `WOOD` / `FIRE` / `EARTH` / `METAL` / `WATER`）で表す。

**相生（しょうしょう）**（母→子の「生む」関係、有向）:

| 母 | 子 |
|----|-----|
| 木 | 火 |
| 火 | 土 |
| 土 | 金 |
| 金 | 水 |
| 水 | 木 |

**相剋（そうこく）**（克つ側→克たれる側、有向）:

| 克つ | 克たれる |
|------|----------|
| 木 | 土 |
| 土 | 水 |
| 水 | 火 |
| 火 | 金 |
| 金 | 木 |

実装は**単一のテーブルまたは関数**に集約し、他モジュールからは参照のみとする。

### 1.3 十干（Jikkan）

10 文字と、五行・陰陽の対応（命理で広く用いられる対応。別表が必要な流派は `ruleset` で上書きテーブルを持てる）。

| 干 | 五行 | 陰陽 |
|----|------|------|
| 甲 | 木 | 陽 |
| 乙 | 木 | 陰 |
| 丙 | 火 | 陽 |
| 丁 | 火 | 陰 |
| 戊 | 土 | 陽 |
| 己 | 土 | 陰 |
| 庚 | 金 | 陽 |
| 辛 | 金 | 陰 |
| 壬 | 水 | 陽 |
| 癸 | 水 | 陰 |

### 1.4 十二支（Junishi）

12 文字と、地支の五行（本気ベースの簡易分類。蔵干は §2 で扱う）。

| 支 | 地支五行（簡易） |
|----|------------------|
| 子 | 水 |
| 丑 | 土 |
| 寅 | 木 |
| 卯 | 木 |
| 辰 | 土 |
| 巳 | 火 |
| 午 | 火 |
| 未 | 土 |
| 申 | 金 |
| 酉 | 金 |
| 戌 | 土 |
| 亥 | 水 |

**Layer 3 実装への注意（位相法）**

- 上表は**地支の本気ベースの簡易五行**であり、Layer 1 のみを参照して**半会・三合会局**等を実装すると不整合になることがある。半会・三合は**中元（季節のピークに相当する蔵干）**の五行など、**蔵干全体の文脈**に引きずられる処理が絡む。
- **対策**: 位相法（Layer 3）は、地支の簡易五行だけでなく **Layer 2 の蔵干テーブル（各地支の初元・中元・本元と採用ルール）を必ず参照**する。本項は実装者向けの依存関係の固定である。

### 1.5 干合（かんごう）ペア（定数）

天干 5 組の合（命理で標準的に列挙されるペア）。**化する五行**まで含めるかは学派により異なるため、**ペアの成立**を Layer 1、**化・虚気発火**は `ruleset`（Layer 2/3）。

| 干 | 干 |
|----|-----|
| 甲 | 己 |
| 乙 | 庚 |
| 丙 | 辛 |
| 丁 | 壬 |
| 戊 | 癸 |

### 1.6 API 上の表現

- 干・支・星・部位は**安定 ID（列挙子または短い code）**を正とし、表示ラベル（漢字・別名）はクライアントの辞書で解決する。
- 五行・陰陽は**全計算の Foreign Key** 的役割を持つ。干支配当テーブルは**単一モジュール**から export する。

## Layer 2：Core Models（静的命式モデル）

「静的」とは**`asOf` によらず**、出生情報と `ruleset` で決まる部分。**`insen` は出生時点の年・月・日の三柱に限定**する（[REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) §6.2）。大運・流年・月運の柱は **Layer 3**（`DynamicTimeline` 等）に置き、**`insen` に折り込まない**（時柱と同様、レスポンスの責務を分離する）。

### 2.1 陰占（いんせん）

**定義**: **年柱・月柱・日柱**の十干十二支（**三柱**）。算命学の宿命算出は「自然のサイクル」に基づき、この三柱のみを命式の器とする。

**時柱・時干について（四柱推命との区別）**

- **時柱（生まれた時刻から定まる柱）は算命学では使用しない**。四柱推命の「四柱（年月日時）」と混同しないこと。
- API の `birthTime`（[REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md)）は**時柱を構成するための入力ではない**。同一暦日内で節入りの前後にいるかを判定し、**日柱・月柱の境界を確定する**ためにのみ用いる（**節入り日**と `birthDate` が一致する場合の必須化など、[OPEN-QUESTIONS.md](./OPEN-QUESTIONS.md) 暦・時刻 **1**）。ドメインモデルに「時干」は出現させない。

**節入り（節気）と万年暦**

- **正統な算命学**（朱学院・高尾学館等の主流系）では、例外なく**節入り（節気）の瞬間**をもって**月柱**を切る。「節入りを使わない系統」は四柱推命の一部・気学の簡易版などに見られ、**本コアが対象とする算命学ドメインでは月柱算法の分岐として想定しない**。
- 各節入りの**瞬間**が月の境界となる。上記のとおり、境界判定に**出生時刻**が必要になる場合はあるが、それはあくまで**暦計算**であり、時柱の生成ではない。
- **API は毎リクエストの天文学的再計算に依存しない**前提とする。`sanmei-core` 同梱の**節入り時刻テーブル**（静的 JSON／SQLite 等）または、検証済みの暦ライブラリ＋**固定エフェメリス**方針のいずれかを [REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) に従い採用する。
- データの出典・版は `rulesetVersion` と紐づける。

**蔵干（ぞうかん）のデータ構造**

- 算命学では蔵干スロットは**初元・中元・本元**で語られることが一般的である。**余気・中気・本気**は主に**四柱推命**側の用語であり、本 API を算命学ドメインに特化させるなら、ドキュメント・監修コミュニケーションでは**初元／中元／本元**に揃える。
- より細かい層（**二十八元**など）を扱う場合は、同じく **`ruleset` のテーブル**で拡張し、本辞書では「スロット列の意味」と「版」を固定する。
- 各地支について、蔵干は**位置意味のある配列／構造体**としてモデル化する（例: 3 スロットなら **インデックス 0,1,2**、またはプロパティ名 **`shoGen` / `chuGen` / `honGen`**（初元・中元・本元のローマ字略））。四柱推命汎用化を最優先しないなら、`earlyQi` / `midQi` / `mainQi` より**監修者との齟齬が少ない**前者を推奨する。
- **どのスロットを「採用（アクティブ）」とするか**は、当該節入りからの**経過日数**（学派により「何日目まで初元か」等の**日数テーブル／深さ**）を入力に、`sect` 付き **IF 規則またはテーブル**で決める。学派間で人体図（例: 胸＝中央の主星）が変わる主因の一つがこの**切替日数定義の差**である。詳細は **監修文書＋`ruleset` データ**に書き、本辞書では**入力型と出力型**のみ固定する。
- **Layer1 → Layer2 の契約（採用方針）**:
  - 経過日数（深さ）は **同一 `context.timeZone` における民用暦日の JDN 差**で定義し、**絶対インスタント間の経過「日」（ミリ秒ベース）では定義しない**。夏時間でその日の実時間が 23h／25h でも、**カレンダー上の「節入りから N 日目」**は暦日差で数える（[IMPLEMENTATION.md](./IMPLEMENTATION.md) §5.0）。
  - 基準節は **月柱計算に用いた `monthSectionEntryAtOrBefore` のエントリと同一**。`rawDelta = JDN_birth - JDN_term`、`displayDepth = rawDelta + SOLAR_TERM_DAY_ZERO_INDEXING`（デフォルト 1）。蔵干テーブルは **`displayDepth`** と照合する（[IMPLEMENTATION.md](./IMPLEMENTATION.md) §5.0）。`rawDelta < 0` は異常。Layer1（三柱解決と同一トランザクション）で算出し Layer2 に渡す。Layer2 は**再実装しない**。
  - **年支・月支・日支**の蔵干採用は、**同じ深さ整数 1 本**を入力とする（立春から年支だけ別数え等は v1 では行わない）。学派差がある場合は **`rulesetVersion` で明示**し、本辞書が普遍の命理定説を保証するものではない。

### 2.2 陽占（ようせん）と人体図

**星の種類（概念）**

- **十大主星**（10）・**十二大従星**（12）: 名称例は公開解説に多数ある（貫索・石門・…／天貴・天恍・…）。実装では**星 ID 列挙**で扱う。

**算出パターン（アルゴリズムの形）**

- 公開教材・実装では、主星・従星は多くの場合 **ルックアップテーブル（行列）** で決まる。
- **典型パターン（例）**（実際の次元は `ruleset` の行列定義に従う）:
  - **十大主星**: **日干**と**比較対象の干**（**年干・月干**および**採用蔵干**（各支の初元・中元・本元から採用された干）など、ルールで定まる干集合）の組をキーに、主星 ID を返す。**時干は含めない**（算命学に時柱がないため）。
  - **十二大従星**: **日干**と**十二支**（**年支・月支・日支**）の組をキーに、従星 ID を返す（実務・教材では年・月・日の支が別枠で語られることが多い）。
- **`mock-v1` の十大主星（エンジン固定の例）**: 主星 5 箇所はいずれも **日干 × 対象の干**の行列参照。**頭＝年干、胸＝月支の採用蔵干、腹＝日支の採用蔵干、右手＝月干、左手＝年支の採用蔵干**（採用蔵干は深さ依存。詳細は [IMPLEMENTATION.md](./IMPLEMENTATION.md) §2）。**監修 `ruleset` では部位と対象の対応が変わりうる**。
- **`mock-v1` 等の十二大従星（検証用規約の例）**: 「日干 × 年支」「日干 × 月支」「日干 × 日支」の **3 回独立ルックアップ**（**干 × 支 → 星 ID**）。**監修 `ruleset` は学派により語り口が異なる**ため、規約は **`rulesetVersion` ごとに文書またはデータで固定**する。
- **行列本体・どの干を読むか**は学派差が大きいため **`ruleset` にバンドル**し、本書は**「行列参照型である」こととキーの型**のみを固定する。

**人体図（部位マッピング）**

- 頭・胸・腹・右手・左手・肩・足など**スロット ID** に星を配置する。部位の**意味**・**読み順**は教材依存。
- **スロット定義（定数テーブル）**は `rulesetVersion` とともに版管理する（複数学派なら `sect` 別テーブル）。

### 2.3 数理法・行動領域（エネルギー／宇宙盤）

**数理法（概念）**

- **集計対象集合**をデータモデルで固定する: 例として「十天干」「採用された蔵干スロット」「年支・月支・日支（およびスコープ内の支）」などを要素とし、**ルールで定めた点数**を合算する**ループ／集計**型。
- **各干・支の重み・掛け算ルール**は公開解説が多様なため、**算法全体を `ruleset` に閉じる**。[REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) の `energyData` と整合させる。

**行動領域（三角形／多角形・宇宙盤）**

- **入力例**: 1〜60 の**干支番号**や、干支に対応する**円周上の角度（0〜360°）**。
- **出力例**: 多角形の**頂点座標列**、行動領域の**面積比**や**重なり**を表すフラグ。
- **角度割当・重なり判定（Intersection）の式**は幾何仕様として `ruleset` または別紙「数理仕様」に書き、本辞書では**I/O の型と用語**に留める。

**Phase L2c の契約（API 固定）**

- **`BaseProfile.energyData`** は **「生まれたままの器」**のポテンシャルのみを表す。**位相法**（半会・三合等）や**虚気**によって**仮想化された干・五行**を入力に含めない。半会後の五行に基づく数理は **Layer3 以降**（将来構想として `shadowEnergyData` 等で分離。コアは戻り値で区切る）とする。詳細は [IMPLEMENTATION.md](./IMPLEMENTATION.md) §2。
- **幾何の API 戻り**は [REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) §6.2 のとおり **極座標の角度＋面積比**に正規化し、**固定丸め**でゴールデン再現性を担保する。

### 2.4 六親法

- 日干を基点に、五行の生剋関係で親族ラベルへ写像する。命理一般では四柱推命が「**十神**」（比肩・劫財・食神…）という語で同種の関係を扱うが、**本 API・本辞書では「十神」という語と ID を使わない**。算命学では**十大主星**（貫索・石門・…）の体系で関係を読むため、データモデル上は **「十大主星（および必要なら五行関係）と六親ラベルの対応」** と記述する。**干／星と六親の対応表は学派・教材により異なりうる**ため、具体的なマッピングは `ruleset`／監修表にのみ書く（本辞書では固定しない）。
- `FamilyNodes` は**クエリしやすいキー**（母・配偶者等）で返す。算命学では**柱・蔵干スロット（初元／中元／本元）への位置**が読みに効くため、詳細は [OPEN-QUESTIONS.md](./OPEN-QUESTIONS.md) §11 でスキーマを監修と確定する。

### 2.5 守護神・忌神（調候）

- 日干・月支（季節＝寒暖燥湿）から、補う五行・避ける五行を返す**マトリクス**を `ruleset` にバンドル。行列の版は監修で固定。

### 2.6 格法・局法（特殊格局）

**定義**: 命式内の五行バランスや星の偏り、特定の組合せによって**格・局**として扱うフラグ。算命学には**完全一気格・従生格**など、学派教材で定義される名称がある（四柱推命の格局分類名である**専旺格・従格・化気**などとは用語体系が異なる）。**本辞書・API では四柱推命の格局名を正規語彙にしない**。実装は監修者が承認した**算命学側の格名**（または内部コード）を `ruleset` で列挙する。

**実装上の位置づけ**

- **格・局**の判定は [IMPLEMENTATION.md](./IMPLEMENTATION.md) §2 のとおり **Layer3b**（位相法・虚気の**後**）。返却フィールド名は `BaseProfile` 内か `interactionRules` かはスキーマ実装で固定するが、**入力**は素の L2 結果と L3a の位相・虚気結果の両方を見る。
- **BaseProfile**（または同等）に、**格・局を表す列挙子＋補助フラグ**を載せる設計が望ましい。`allowGohouInKaku` が true のとき**位相法を格判定に織り込む**か等は [SECT-RULESET-MATRIX.template.md](./SECT-RULESET-MATRIX.template.md) と `ruleset` で定義。
- 判定条件（五行カウント閾値、星の集合パターン等）は**学派差が大きい**ため本辞書では**カテゴリ名と出力スロット**のみ示し、真理値表は監修データへ。

## Layer 3：Dynamic Interactions（動的演算と相互作用）

入力は **`context.asOf`**・**大運サイクル**・**流年流月**、および命式＋運の**干支関係**。

### 3.1 大運・年運・月運

- **大運**: 10 年単位の柱と星の列。順逆行は `gender` と `ruleset`（[OPEN-QUESTIONS.md](./OPEN-QUESTIONS.md)）。
- **立運年齢（何歳運）**: 誕生日から、順行なら**次の節入り**まで、逆行なら**前の節入り**までの**日数を 3 で割った商**（および学派による端数処理）で求めるのが一般的なパターン。**余りの扱い（切り捨て・四捨五入等）**や**起算日の数え方**は学派間で明確に異なりうるため、**人体図・大運開始年齢の差異の第二の主因**となる。算法は `ruleset` に閉じ、[SECT-RULESET-MATRIX.template.md](./SECT-RULESET-MATRIX.template.md) で版管理する。
- **年運・月運**: `asOf` に対応する干支（＋星）。節入りテーブルと整合させる。

### 3.2 天中殺（宿命・年運・大運）

**API 上の配置（採用）**

- **出生で凍結する宿命系**（生涯変わらない宿命天中殺・当該する異常干支フラグ）: `BaseProfile.destinyBugs`（Phase L2c）にのみ載せる。
- **年運・大運・スライド・`asOf` で変わる稼働**: `DynamicTimeline.tenchuSatsuStatus`（[REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) §6.3）。**混在させない**（静的／動的の二重管理を防ぐ）。

**フラグの概念**

- **宿命天中殺**: 出生情報から決まる異常干支・天中殺系フラグ（製品定義に従う）。
- **年運天中殺**: 通常 12 年周期に関する説明が一般解説にある。期間・起算は `ruleset`。
- **大運天中殺**: より長期のブロックとして説明されることがある。`isShifted` 等の**確変**表現は数値・フラグに限定（ライフコーチ文は返さない）。

**スライド（移動）**

- 大運天中殺の**移動**は算命学実装の**最大級の難所**の一つである。例として、**第三旬の異常干支**に起因するスライド、**大運同士の干合**に起因するスライドなどが語られるが、**どこまで波及するか**は熟練鑑定士の間でも意見が分かれうる領域である。
- 要件上、位相法の成立により**期間ブロックが前後に拡張／移動する**ような例外処理が想定される。**コアをハードコードで埋めず**、`ruleset` 側の **JSON 等による DSL（ドメイン固有言語）**に逃がし、バージョンで固定する設計が望ましい。
- **具体的な条件式・再帰／伝播の定義**は監修確定後の `ruleset` 仕様とする（[OPEN-QUESTIONS.md](./OPEN-QUESTIONS.md) §7）。

### 3.3 位相法（いそうほう）

**タクソノミ（例）**

学派により名称・分類が異なる。API では `kind` 列挙子で正規化する。例:

- 干合、支合、半会・三合、方三位、刑、害、**対冲（たいちゅう）**（四柱推命で単に「冲」と書かれることがある関係。API の `kind` は算命学に合わせ **`TAICHU` / `taiChu`** 等で正規化）、納音、律音、天剋地冲 など。

**競合解決（コンフリクト）**

- 半会・三合等の判定では §1.4 の注意どおり **Layer 2 蔵干テーブル**を参照すること。
- 複数位相が同時成立しうる。**合法（まとまる）と散法（対冲などで離れる）が同時**などの優先は**学派依存**。
- バックエンドでは [REQUIREMENTS-v1.1.md](./REQUIREMENTS-v1.1.md) の **Priority Resolver** により `strength`・`priorityTier`・`resolved` / `suppressed` を返す。**「散法優先」等のタイブレーク順は `rulesetVersion` ごとに文書化**する。
- `InteractionRules` の各現象に **優先度・スコア**を載せる設計は、UI と LLM の説明責任の両方に効く。

### 3.4 虚気（きょき）

**干合**: Layer 1 のペア定数を参照。

**発火条件（Trigger）**

- 例: 命式内または運との間で干合が成立し、かつ**月支の五行条件**を満たす、**隣接のみ有効**、等。**厳密な条件は `ruleset` の真理値表**（[OPEN-QUESTIONS.md](./OPEN-QUESTIONS.md) §8）。

**状態とデータの持ち方**

- **BaseProfile を上書きしない**。虚気成立時は **`shadowYousen`（裏の十大主星）** または別名の **VirtualProfile** 相当オブジェクトを**別フィールド**で返す。元の陽占は検証・比較のため保持する。
- **数理（エネルギー）との関係**: [IMPLEMENTATION.md](./IMPLEMENTATION.md) §2 のとおり、**L2c の `energyData` は素の器のみ**。位相・虚気を加味したエネルギーは **Layer3 以降**で別フィールド（例: `shadowEnergyData`）を検討し、`energyData` と混在させない。

## 学派差（算命学における本質的差分）とデータ倫理（横断）

**算命学正統各派に共通すること**

- **月柱は節入り（節気）で切る**。朱学院・高尾学館のような主流算命学において「節入りを使わず月柱を決める」分岐は、本コアのドメイン前提からは**除外**する（他術種への汎用は別パッケージの責務）。

**学派間で人体図・大運などがずれる主な理由（実装でマトリクス化する）**

1. **蔵干（初元・中元・本元）の切替日数**  
   節入りから何日目までを初元とみなし、何日目から中元・本元に移るかの**日数テーブル（深さ）**が学派間で異なる。これにより**採用蔵干**が変わり、陽占（例: 人体図の中央＝胸の主星）が変わる。
2. **大運の立運年齢の端数処理**  
   節入りまで（順行／逆行）の**日数÷3**の**余りの扱い**（切り捨て・四捨五入等）および**起算の数え方**が学派間で異なり、**何歳運から始まるか**が変わる。

二次情報では「節入り日を含める／含めない」が違いのように読めることがあるが、**正統算命学では月の境界は節入りに統一**され、上記 1・2 が結果差の説明として本質的である。**実装は監修データと `sect` マトリクス**に従う（[SECT-RULESET-MATRIX.template.md](./SECT-RULESET-MATRIX.template.md)）。

**データ倫理**

- 他社無料占いフォーム等の**スクレイピングによるルール抽出は禁止**。契約・監修・自前データで実装する。
